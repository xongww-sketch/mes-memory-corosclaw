### 我的分析与建议

#### 中框报废后SN处理：闭环旧SN、生成新SN
- **行业标准（富士康/立讯/歌尔）**：组件可复用，SN不复用
- ISO 13485 / IATF 16949 追溯性原则：SN是物理个体身份证，不可转移
- SN复用的问题：追溯断裂、售后混乱、数据不一致、质量分析失效
- **熊旺认可这个方向**（"你的逻辑看着好像没问题"）

#### SN状态扩展（最小改动方案）
在原有 0/1/2/3 基础上新增：
- **4 = 维修中**
- **5 = 报废**
- **6 = 丢失**
- 作废不需要单独状态，复用 3（已失效）

#### 维修改造为"进站-出站"模型（重大改进）

熊旺提出关键意见：
1. 作废和维修应该在同一个页面做
2. 维修结束应该代表出维修站
3. 维修站内可能一直有SN没出站
4. 进维修尽量统一录入，表示进站

**新方案：维修工位操作台**

进站：任何工序发现不良 → 扫SN → 确认送维修 → status=4（维修中）→ repairingQty+1

维修中列表：显示所有status=4的SN，含滞留时长预警

出站三种结果：
- **修复**：填维修记录 → status恢复 → repairingQty-1 → 回产线
- **报废**：填报废原因 → 解绑 → status=5 → repairingQty-1, scrapQty+1, pendingReplenishQty+1
- **作废**：填作废原因 → status=3 → repairingQty-1

工序扫码校验：
- 允许扫码：status IN (1, 2)
- 拒绝扫码：status IN (0, 3, 4, 5, 6)，各给不同提示

#### 丢码处理方案
- 三层防线：预防（工序校验一进一出）→ 发现（异常SN 48h预警）→ 处置（标记丢失→缓冲期→报废）
- 丢失SN 24h缓冲期（可能找回来）
- status=6（丢失），找回可恢复，超时自动转报废

### 与之前F3设计的变化
| 之前F3 | 现在方案 |
|--------|---------|
| 入维修/出维修/直接报废三个操作 | 进站 + 出站三选一（修复/报废/作废） |
| 报废是维修页的一个分支 | 报废是出站结果之一 |
| 没有进站概念 | 进站=扫SN确认送维修，status=4 |
| 没有维修中列表 | 新增维修中SN列表Tab |
