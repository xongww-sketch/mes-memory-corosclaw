### 系统数据流梳理（已彻底搞清）

**渠道数据来源**：
- 国内：`warehouseDelivery.public.mes_delivery_order_details`（retroid + channel + create_time）
- 国外：OMS MongoDB `snoutbounds` 集合（sn + channel + outboundTime）

**现有定时任务 `get_channel`**（resT6rwMQjZMNQXc，每天 3 点）：
1. 从国内/国外源拉取有渠道的 SN
2. 写入 `mes_daily_data.delivery_detail`（建立 SN↔渠道关系）
3. join `production_detail` → 插入 `daily_info`（含 channel）

**旧 MES 渠道回写**：
- `8auEQ5ousSKMg5wd`（Works 渠道同步）+ `sZ1WSJAx0jJXGQeO`（经销商渠道同步）

### 关键约束

- **跨库不能直接 JOIN**：warehouseDelivery 库 vs mes 库 → 必须分步查询
- **agent 沙箱无 DB 直连**：所有修复必须在 n8n 内执行
- **mes_info 只拉最新 createtime 的行** → 旧空行无需删除

### 经验教训
1. n8n postgres insert 节点导入后会自动把主键id加进映射且值=0 → 必须把id+自增列标 removed:true，否则整批违反主键约束回滚
2. 汇总/报告节点要读"真实写入结果"而非"计划数据"，否则报虚假成功数误导排查
3. splitInBatches done输出不携带全量累积数据 → 改批量集合操作时用 $('节点').all() 跨节点引用，或用Merge汇合保时序
4. n8n并行分支时序不保证，需要"先写后报"时用Merge汇合而非并行
5. SQLServer节点 onError:continueErrorOutput 但错误口没接节点 → 报 Cannot read properties of undefined(reading 'error')，改 continueRegularOutput
