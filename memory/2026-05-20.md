## FPC绑定中框校验SQL优化

### 背景
熊旺在优化 `fpc_bind_to_sn_v3_final.json` 工作流，核心需求是**确保 SN 与中框的一对一绑定关系**：一个 SN 只能绑一个中框，一个中框也只能绑一个 SN。

### 校验规则（最终确认）
1. **SN 维度**：如果 SN 已绑定中框且未解绑 → 拦截，报错"SN已绑定中框"；已解绑则允许重新绑定
2. **中框 维度**：如果中框已绑定某个 SN 且未解绑 → 拦截，报错"中框已绑定SN"；已解绑所有 SN 则允许重新绑定
3. **核心难点**：`machine_data` 表中绑定和解绑记录频繁交替，SQL 必须正确追溯"当前是否处于绑定状态"，而非简单看是否有绑定记录

### 问题发现
熊旺提供了真实数据（`machine_data111.csv`），发现之前写的校验 SQL 存在**严重缺陷**：频繁解绑/重绑场景下，同一 SN 仍然可以绑定两个不同的中框。从 `createtime` 可以看出 145 工序跑了两遍，校验未能拦截。

### 根因分析
原 SQL 使用 `NOT EXISTS` 查解绑记录，但解绑判断逻辑不完整——仅通过 `machine_data_id_old` 或 `unbind_id` 关联，**未能覆盖所有解绑场景的追溯链路**，导致某些解绑操作未被正确识别，从而让"已绑定"的 SN 被误判为"未绑定"。
