### WiFi MAC (procedureDetailId=148) 覆盖缺陷修复

**问题**：同一 retroid 有两条 procedureDetailId=148 记录（一条有 mac 值、一条为空），createtime 几乎同时无法判断先后，空值可能覆盖有效 mac 值。

**方案演进**：
1. **方案 A（被动防护）**：非空保护——只有当记录确实有值时才写入，空值不覆盖已取到的值
2. **方案 B（主动锁定，熊旺倾向）**：优先级合并——区分"解绑"和"有效绑定"两类记录，**只要存在任意一条有 mac 的有效绑定，就锁定取值，解绑记录和空记录都无法冲掉**

**最终方案**：引入 `_locked` 内部标记的"有效绑定优先锁定"策略。同时加固了 5 个同类风险字段：
- 148 (WiFi MAC)
- 39 (UUID)
- 31 (Battery)
- 50 (Screen)
- 59 (Remark)

## 🍎 Apple MFI 工作流全链路分析

**触发**：熊旺要求分析 Apple 项目下所有工作流中 `applemfi` 表每个字段的插入逻辑，基于 n8n 最新工作流。

### 项目概况
- Apple 项目下共 **6 个工作流**，其中 2 个 ACTIVE（新 MES 版本）
- 核心落库表：`machine_data`（applemfi 数据最终写入此表）

### 数据流全景
```
产线扫码 → 申请 Apple Token → CRC16 转换 → Code 节点组装 → Postgres 插入 machine_data
```

### 关键字段来源
- **Token 获取**：HTTP Request 节点调用 Apple 接口申请 token
- **Token 状态流转**：初始 → status=2（已使用），通过"查询是否有可用 token"节点判断
- **CRC16 转换**：Code 节点处理 CRC16 校验码转换
- **字段组装**：多个 Code 节点（Code/Code1/Code2/Code3）分别处理不同字段组的映射
- **最终写入**：Postgres 节点插入 `machine_data` 表

### 关键发现
- 新旧 MES 两套工作流并存，都在往 `machine_data` 表写入
- applemfi 表本身不直接存储数据，而是通过 `machine_data` 的 data 字段间接存储
- Token 申请 → 使用 → 状态更新的完整链路已理清
