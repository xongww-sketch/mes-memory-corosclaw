## 主题一：n8n 工作流 Bug 修复 — 一个 SN 绑定多个中框

### 问题描述
- **工作流**：`fpc_bind_to_sn`（ID: 6ECMz9f2gyozcJyz），API `POST /procedure/sn-bind`
- **Bug**：一个 SN 可以绑定两个不同的中框（中框145工序），这是业务上不允许的
- **根因**：原工作流只做"该 SN + 该中框"的查重，不做"该 SN 是否已绑定任何其他中框"的检查

### 工作流深度分析
- `machine_data` 表是**只增不删**的流水表
- 绑定记录：`procedure_detail_id = 145`，`data.bind_id` = 中框ID
- 解绑记录：`procedure_detail_id = 62`，`data.unbind_id` = 被解绑的中框ID
- 解绑后可以重新绑定（新的 bind 记录）
- 相关子工作流：`binding_materials_before_processing`（解绑）、`get_Package_bind_status`（查询绑定状态）

### 修复方案
在原流程「查询SN和FPC绑定信息1」之前，新增**防一SN多中框**校验节点（3个节点）：

1. **查询SN是否已绑定其他中框** — SQL 检查是否存在有效的绑定记录（即没有对应解绑记录）
2. **If5 判断** — 如果有未解绑的绑定 → 返回错误（该SN已绑定中框XXX，请先解绑）
3. 校验通过 → 继续原流程

修复后的工作流 JSON 已上传到熊旺飞书云空间：`fpc_bind_to_sn_fixed.json`

### SQL 校验逻辑迭代（重要教训）

#### 第一版 SQL（有 bug ❌）
```sql
NOT EXISTS (
  SELECT 1 FROM mes.machine_data md2
  WHERE md2.procedure_detail_id = 62
  AND md2.retroid_id = md.retroid_id
  AND md2.data->>'unbind_id' = md.data->>'bind_id'
  AND md2.createtime > md.createtime   -- ← 问题在这里
)
```

#### 问题暴露
熊旺用真实数据测试发现：**绑定记录和解绑记录的 `createtime` 完全相同**（同一次操作生成的两条记录），`>` 比较符导致解绑记录无法"覆盖"绑定记录，查询仍返回旧绑定数据。

#### 修正版 SQL（✅）
```sql
WITH latest_binding AS (
    SELECT md.*
    FROM mes.machine_data md
    WHERE md.procedure_detail_id = 145
      AND md.retroid_id = 830214
      AND md.data->>'bind_id' IS NOT NULL
      AND NOT EXISTS (
          SELECT 1 FROM mes.machine_data md2
          WHERE md2.procedure_detail_id = 62
            AND md2.retroid_id = md.retroid_id
            AND md2.data->>'unbind_id' = md.data->>'bind_id'
            AND md2.createtime >= md.createtime   -- ← 改为 >=
      )
    ORDER BY md.createtime DESC
    LIMIT 1
)
SELECT * FROM latest_binding WHERE bind_id != '目标中框ID';
```

**关键修正**：`createtime >` → `createtime >=`，解决同时间戳场景下解绑不生效的问题。

### 关键教训
1. **同时间戳场景必须考虑**：绑定和解绑是同一事务生成的两条记录，`createtime` 完全相等，不能用 `>` 判断先后
2. **只增不删表的查询逻辑**：必须用 `NOT EXISTS + 时间戳 >=` 的方式找"最新有效记录"
3. **SQL 必须在真实数据上验证**：逻辑上正确的 SQL 在边界场景可能失效
