## 📺 树莓派组装看板 SSE 实时推送故障排查（06:40 - 08:22）

### 问题描述
组装看板在 Mac 上通过工厂 WiFi 访问时数据实时更新，但在树莓派（NanoPi R76S）上用 Chrome kiosk 模式打开同一页面时，SSE 推送不工作，数据卡死不变化。

### 技术环境
- **树莓派**：NanoPi R76S，Chromium 130.0.6723.58
- **看板 URL**：`https://mes.factory.coros.team/#/dashboard/assembly/workOrder?lineName=组装K线`
- **后端 SSE 端点**：`/server/joborder-stats/events/assembly-dashboard/joborder-stats?lineName=组装K线`
- **SSE 特征**：`retry: 3000`，事件类型 `connection` → `init` → 数据事件
- **对比环境**：Mac 同 WiFi 正常，DNS 解析一致（192.168.128.13）

### 排查过程与发现

1. **SSE 连接验证**：curl 直接请求 SSE 端点 ✅ 成功返回 `connection` + `init` 事件，初始数据完整
2. **Chrome 启动参数**：已加 `--disable-background-timer-throttling` `--disable-renderer-backgrounding` — 无效
3. **Chrome 远程调试**：port 9222 可访问，但 CDP WebSocket 探测遇到 SSL handshake failed（net_error -100）
4. **Python WebSocket 探针**：尝试直接连接 Chrome DevTools WebSocket 调试 SSE 连接状态，因 SSH 管道冲突（Device or resource busy）受阻
5. **Chrome 日志分析**：`--enable-logging=stderr --v=1` + grep SSE/EventSource/error — 输出显示 SSL handshake failed 反复出现
6. **熊旺截图**：页面时间戳在跳动（说明 JS 在运行），但看板数据卡死不动

### 初步判断
- SSE 连接最初能建立（curl 验证通过），但 Chrome 内的 EventSource 连接可能在建立后断开或不再接收数据
- SSL handshake failed 错误可能与证书链或 Chrome 版本有关（Chromium 130 vs Mac Safari/Chrome 版本差异）
- 树莓派 DNS 解析与 Mac 一致，排除 DNS 问题
- 网络层无问题（curl 能拿到完整 SSE 数据）

### 待排查方向
- [ ] Chromium 130 的 EventSource 对 SSE retry 机制是否有已知 bug
- [ ] SSL 证书在 Chromium 130 上是否被标记为不可信（handshake failed 根因）
- [ ] 尝试 HTTP 直接访问（不走 HTTPS）看是否是 SSL 层问题
- [ ] 检查树莓派系统时间是否与服务器同步（SSL 时间敏感）
