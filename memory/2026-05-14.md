## 主题一：Agent 知识库适配方案（新方向）

**背景**：MES 架构文档体系（5份文档）已完成，熊旺提出下一步——让知识库能被任何 agent/AI 即插即用地使用。

### 核心需求
- 部门同事用任何 agent（OpenClaw、Claude Code、Codex、Cursor 等）都能直接问 MES 业务/技术问题
- 知识库要成为**跨 agent 通用的知识底座**

### 核心约束（熊旺明确）
1. **文件结构神圣不可侵犯**：目录结构、文件命名、章节框架由人类规划，agent 只能在指定"内容槽位"内增改删
2. **共建共维 ≠ 自由发挥**：所有可写入位置、可更新字段、可追加小节必须明确定义
3. **写入路径单一可审计**：Agent 不走飞书写权限，统一走「向代码仓库提 PR → 守护人审核 → 合并」

### 基础设施确认
- **Source of Truth**：公司 GitLab `http://git.coros.team/`（内网）
- **外部镜像**：GitHub 仓库供外部 agent 拉取
- **守护人**：熊旺（maintainer），所有 PR 由其审核合并
- **同步频率**：飞书 → Git 每日一次（凌晨定时任务）
- **可用服务器**：`192.168.64.54`（已部署 Docker / OpenClaw / Hermes / 飞书机器人通道）

## 主题二：定时记忆同步任务修复

### 问题
熊旺发现 cron 定时任务（每日 21:00）存在两个问题：
1. **读不到当天对话**：extract-daily-memory.py 从 session 文件提取，但 session 文件的时间戳和实际对话时间有偏差
2. **直接搬运原文**：之前脚本只是把原始对话追加到 memory 文件，没有经过整理归纳

### 熊旺的要求（最终明确）
> "我需要的是你读到每天的聊天记录后，通过和你对mes的所有记忆，宏观整理一下再推到github，相当于整理记忆和文档，不是直接把你收集到的消息不经过处理推过去"

### 本次执行方式
- 手动执行 extract-daily-memory.py 提取原始对话
- **按主题分段整理**，提炼关键决策和结论，不搬原文
- 标注方案变更和待办事项
- 有重大决策时同步更新 MEMORY.md
- 最后通过 memory-sync.sh 推送到 GitHub

### ⚠️ 遗留问题
- cron 定时任务的时间问题（21:00 UTC = 次日 05:00 CST）可能导致"当天"对话提取不完整
- 需要确认 cron 的时区配置，或调整执行时间
