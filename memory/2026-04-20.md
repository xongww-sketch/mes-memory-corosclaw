## 熊旺：组装段工单结单/补料/维修报废/看板需求收口

今天与熊旺集中讨论了组装段相关需求，已形成较完整的统一业务方案方向，等待明天上午继续产出正式需求文档。

### 已确认/基本认可的核心方向

1. **工单结单方式调整**
   - 从“按SN维度结单”改为“按工单维度结单”。
   - 工单必须与**计划数量对齐**，报废不能直接冲抵计划。
   - 结单逻辑趋向自动结单，不希望增加人工“待结单”动作。
   - 当前收口后的主状态建议：
     - 未开始
     - 生产中
     - 已结单
   - 自动结单条件建议口径：
     - goodQty = planQty
     - repairingQty = 0
     - pendingReplenishQty = 0
     - wipQty = 0

2. **补料机制**
   - 报废/损耗后必须补齐计划数。
   - 不建议再新开独立正式补料工单，**补料SN应归属原工单**。
   - 当前阶段不依赖审批流，先走“人工确认生成补料SN”的轻量方案。
   - 报废是系统可感知损耗；SN物理丢失系统不可自动识别，必须走**人工损耗登记**。
   - pendingReplenishQty 作为待补料数量。

3. **维修/报废工序设计方向**
   - 维修工序：独立中间态工序，建议**扫码进维修 / 扫码出维修**。
   - 报废工序：独立终态工序，建议**扫码确认报废**。
   - 不能只维护工单层面的维修数/报废数，必须至少保留**SN级事件记录**，工单层做汇总统计。
   - 不建议构建过于复杂的SN全局状态机，建议“SN级事件记录 + 少量业务标志 + 工单统计表”。

4. **组装看板方向**
   - 看板显示对象不是单纯“当天计划工单”，而是**活跃工单池**。
   - 活跃工单池包括：
     - 今日计划工单
     - 历史未结单工单（遗留）
     - 临时插单且已投产工单
     - 当前存在在制/维修中/待补料等状态的工单
   - 核心原则：**未结单不能下看板**。
   - 看板核心字段建议包括：
     - 工单号
     - 分类（今日计划/遗留/插单）
     - 状态
     - 计划数
     - 良品数
     - 维修中
     - 报废
     - 待补料
     - 在制
     - 剩余
     - 当前主阶段
     - 四工序进度
   - 四工序必须体现“阶段到达数量”，而不是单纯累计扫码次数。
   - 四工序为：
     - 上架老化
     - A/B壳点胶
     - 功耗静置
     - 外观产出
   - 建议展示为四工序到达SN去重数量，例如：100 / 92 / 80 / 75
   - 并结合相邻工序差值，辅助判断阶段堆积与卡点。

5. **数量模型收口**
   - 建议核心统计字段只保留少量：
     - planQty
     - goodQty
     - repairingQty
     - scrapQty
     - replenishQty
     - pendingReplenishQty
     - wipQty
   - 以及四工序到达数。
   - remainQty = planQty - goodQty
   - wipQty = actualInputQty - goodQty - scrapQty - repairingQty
   - actualInputQty = 初始SN数 + replenishQty

6. **数据模型建议**
   - 不建议把所有统计字段硬塞进工单主表。
   - 建议：工单主表 + 工单统计扩展表（如 joborder_stats / joborder_progress）+ 维修/报废/补料事件记录表。
   - 原因：保持主表干净、利于看板查询、后续扩展容易、能保留追溯。

7. **历史工单问题（重点）**
   - 熊旺明确提到：历史上很多工单长期未结单，原因是原来SN丢码/报废后，产线会**新开工单生成新SN**，物理上补齐原工单，但系统中两个工单完全无关。
   - 这会导致：
     - 原工单长期未结单
     - 新工单也可能定义混乱
     - 系统和物理产出不一致
   - 当前结论：
     - 新规则上线后，未来不应再用“新工单补原工单”的方式
     - 历史数据需要单独设计**历史工单补料挂账修复方案**
     - 允许人工补录“原工单-补料来源工单”的关系用于清理历史未结单

8. **工单类型治理（重点）**
   - 熊旺明确指出当前工单类型使用混乱：
     - 组装
     - 包装
     - 维修
     - 售后
     - 反投
   - 问题点：
     - 有些补料工单被建成组装类型
     - 丢失SN补量也被建成组装类型
     - 维修类型被用于产线维修
     - 反投用于售后机型、7天无理由、返修/寄修要走组装段测试的场景
     - 试产没有工单类型
   - 当前收口方向：
     - 主工单类型建议收敛为：
       - 组装
       - 包装
       - 返投
     - 维修工单类型仅在确有“独立维修业务”时保留，不应用于产线维修
     - 售后不建议单独作为主工单类型，更适合作为来源属性，回产线的售后场景并入返投
     - 试产需要补充定义：可考虑做成工单属性，或视业务差异决定是否单独成类型
     - 补料不应再作为独立工单类型或借组装类型乱建
