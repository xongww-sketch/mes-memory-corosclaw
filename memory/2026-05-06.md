## Hermes Agent 修复（上午）

### 问题
Hermes 在 192.168.64.54 上模型调用 403 报错，无法响应飞书消息。

### 根因
CCSwitch 平台模型名从 `opus-4-6` 改成了 `claude-opus-4-6`，Hermes 配置未同步更新。

### 修复过程
1. SSH 密钥配置（之前没有）— 生成 ed25519 key，熊旺手动加到 authorized_keys
2. 修改了 3 处配置：
   - `/opt/hermes/config/config.yaml` — gateway 配置
   - `/opt/hermes/config/.env` — 环境变量
   - `/opt/data/config.yaml` — **实际加载的运行时配置**（HERMES_HOME=/opt/data）
3. 所有配置改为：
   - `model: claude-opus-4-6`（不是 opus-4-6）
   - `provider: anthropic`（不是 custom）
   - `base_url: http://llm.apps.coros.team`（不带 /v1）
   - 新增 `CLAUDE_CODE_DISABLE_EXPERIMENTAL_BETAS=1`
4. `docker compose up -d hermes` 重建容器（docker restart 不会重载 .env）

### 教训
- Hermes 有 3 层配置文件，实际加载的是 `/opt/data/config.yaml`
- docker restart ≠ 重载 .env，必须 docker compose up -d 重建
- CCSwitch 截图是关键参考：API格式、认证方式、模型名

### n8n 工作流分析

**repair_message（50 nodes）— 售后维修/绑定后维修**
- 校验链：参数完整 → UUID位数(30) → UUID查绑定(machine_data) → 获取SN → 维修工序存在(id=66) → SN前3位匹配机型 → 工单颜色一致 → 简单/复杂维修分支
- 复杂维修：先查绑定数据 → 解绑 → 插入维修记录
- 写入 machine_data（procedure_detail_id=66）

**repair_message_line（68 nodes）— 产线维修/绑定前维修**
- 校验链：参数 → UUID绑定状态 → SN有效性 → 重复绑定检查 → SN使用检查 → SN属于工单 → 机型匹配 → MCU维修分支 → 电池解绑分支
- MCU维修：额外的uuid_old处理
- 电池解绑：查retroid_id → 查电池绑定(pd_id=31) → 查是否已解绑(31,62) → 执行解绑

**get_repair_message（13 nodes）— 查询维修记录**
- 按UUID查 procedure_detail_id=66 的 machine_data
- 返回：产品类型、不良现象、维修信息、待机电流、MCU状态、SN、员工工号、工单号

### 关键数据模型
- 核心表：mes.machine_data（JSONB data字段）
- 维修工序：procedure_detail_id = 66
- 电池绑定/解绑：procedure_detail_id = 31 / 62
- SN与UUID绑定 / 物料解除绑定：通过 procedure_name 查 procedure_detail
- SN前3位→机型映射表（M1D→M04, B20→B20, W335→W35 等19个映射）
