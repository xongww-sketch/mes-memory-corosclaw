# 2026-05-08 Daily Notes

## 作废 vs 报废 概念重构（重大方案变更）

### 熊旺的关键洞察
熊旺提出：**报废 ≠ SN报废，报废 = 物料报废**。SN层面只有一个动作——**作废**。

| 层面 | 动作 | 含义 |
|------|------|------|
| **SN层面** | 作废（status→3） | 这个编号不能再用了 |
| **物料层面** | 报废 | 中框/电池/主板等物理器件坏了（人工登记，不在MES做） |

### 新方案：SN统一作废
不管什么原因导致SN不能继续用，统一叫**作废**：
- 丢码找不到了 → SN作废
- 中框坏了换新 → SN作废
- 误生成/多生成 → SN作废

**好处：**
1. 直接用现有 `status=3（已失效）`，不需要新增 status=5（报废）
2. 所有30+个n8n工作流已经在排除 status=3 的SN，改动最小
3. 概念更简洁：SN只有 有效/无效 两种状态

### 工单统计表字段调整
```
scrapQty → 改名叫 voidQty（作废数）更准确
pendingReplenishQty → 待补料数（作废+1就补料+1）
```

### status=4（维修中）保留
维修中是**中间态**，不是终态，需要单独状态：
- SN进站维修 → status=4, repairingQty+1
- SN出站（修复/作废）→ status恢复或→3, repairingQty-1

## 工单结单逻辑重新梳理

### 工单核心等式
```
planQty = saleableGoodQty + sampleQty  ← 结单时必须成立
```

每个SN最终只有4种归宿：
| 归宿 | 计入 | 说明 |
|------|------|------|
| ✅ 良品完工 | saleableGoodQty | 正常交付 |
| 🎁 样机领走 | sampleQty | 特殊交付 |
| 💀 作废 → 补料新SN → 最终良品 | 新SN计入saleableGoodQty | 旧SN作废，新SN补上 |
| 🔍 丢失 → 补料新SN → 最终良品 | 新SN计入saleableGoodQty | 旧SN作废，新SN补上 |

### 自动关单条件（更新后）
```
completionQty >= planQty
且 repairingQty = 0
且 pendingReplenishQty = 0
其中：completionQty = saleableGoodQty + sampleQty
```

### "丢失"状态讨论
熊旺质疑"丢失"状态的必要性。结论：**一期不做独立的"丢失"状态**，丢失场景直接走作废流程。理由：
- 产线上SN找不到，大多数是真的找不到了
- "丢失"本质上是"报废前的缓冲期"，实际价值有限
- 简化方案：找不到就作废→补料

### retroid.status 使用模式（关键发现）
**绝大多数工作流用同一种模式校验SN是否有效：**
```sql
r.status in ('1','2')    -- 60+次，30+个工作流
```

| 模式 | 次数 | 含义 | 涉及工作流数 |
|------|------|------|------------|
| `status in ('1','2')` | 60+次 | SN有效才允许操作 | 30+个 |
| `status != 3` 或 `status <> 3` | 17次 | 排除已失效的SN | 8个 |
| `status = 0` | 19次 | 查反投机SN | 2个 |
| `status = 1` | 22次 | 查已生成未绑定token | 7个 |
| **WRITE: SET status = 0** | 1次 | 反投机操作 | reassy_blind_api |

**关键结论：** 加 status=4（维修中）后，所有 `status in ('1','2')` 的校验**不需要改**——维修中的SN本来就不应该被正常工序扫码。

## 场景穷举与解决方案

### 无动态SN
| 场景 | 有无实物 | 处理 | 是否补料 |
|------|---------|------|---------|
| 误生成/多生成 | 无 | 作废（status→3） | ❌ |
| SN已生成，中框未上线 | 有（空白中框） | 正常上线或作废 | ❌ |

### 有动态SN
| 场景 | 处理 | 是否补料 |
|------|------|---------|
| 正常流转 | 继续工序 | — |
| 进站维修 | status→4, repairingQty+1 | — |
| 维修后修复 | status恢复, repairingQty-1 | — |
| 维修后作废 | status→3, 解绑, voidQty+1, pendingReplenishQty+1 | ✅ |
| 丢码找不到 | 作废（status→3） | ✅ |
