## 📊 FPY 两级良率体系讨论（06:41 UTC / 14:41 CST）

**发起人**：熊旺
**问题**：上位机查出不良 → 复测仍不良 → 发现是排线松/主板螺丝没打紧 → 技术员处理后复测通过。这种机子该不该被认定为不良品？

### 核心结论：良率不是"一个数字"，而是"两个数字"

成熟 MES 的标准解法：**同时记录 FPY（一次通过率）和最终良率**，不纠结"到底算不算不良"。

| 指标 | 含义 | 该场景的归属 |
|------|------|-------------|
| FPY（First Pass Yield）| 第一次测试就 OK，中途没被 NG 过 | ❌ 算不良（FPY 掉了）|
| 最终良率（Final Yield）| 经过处理后最终 OK 出货 | ✅ 算良品（修好出货了）|

### 关键洞察：我们缺的是"现场返修"这一档

现在产线的处置能力只有两档：
- **送修**（走维修工站，改 SN status=4）
- **作废**（status=3）

但"排线松了拧一下就好"这种**轻微返修**，不应该走维修工站——它应该记为"现场返修（On-line Rework）"，不改 SN 主状态。

### 落地方案（贴合现有 MES 架构）

**不改 SN 主状态，只加记录层**：

```
测试记录（test_record）：
  SN + 工序 + result(NG/PASS) + defect_code + seq 序号

返修记录（rework_record）：
  SN + defect原因 + 处理动作 + disposition(现场返修/离线维修/报废)
```

- 上位机 NG → 写 test_record(result=NG)，SN 状态不变
- 技术员处理 → 写 rework_record(disposition=现场返修)
- 复测 PASS → 写 test_record(result=PASS)
- FPY 报表 = 从 test_record 算（有过 NG 的都算首次不良）
- 只有走送修工站/作废才动 SN 主状态

**关键优势**：不污染 SN 主状态机，不影响现有 322 个 n8n 工作流。

2. **上位机测试数据回传**：上位机是每次 NG/PASS 都实时写进 MES，还是只有最终结果进？这决定 FPY 能不能自动算。
