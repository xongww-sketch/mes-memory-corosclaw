**核心问题**：Code 节点中 `procedureDetailId=148`（WiFi MAC）存在覆盖式赋值缺陷

### 问题诊断

当同一 retroid 有两条 `procedure_detail_id=148` 记录时（一条有 mac 值，一条为空），原 Code 节点用直接覆盖赋值：
```js
results[retroidId].wifi_mac = isUnbound ? null : record.json.data.mac;
```
**后果**：有值记录先来、空值记录后到时，mac 被覆盖成 null。SQL 查询无 `ORDER BY` 保证顺序，结果不稳定。

### 用户确认的业务前提

- 两条 148 记录的 createtime 几乎同时，**无法通过时间判断先后**
- 148 是**同一绑定信息**，不是绑了又解的重复绑
- 策略：**哪条有 mac 就取哪条**

### 交付方案（两版迭代）

**方案一：非空保护（setIfPresent）**
- 所有字段改为"仅本条有有效值时才写入"
- 空值绝不覆盖已取到的值
- 解绑（unbound=true）语义保留，仍置 null
- 顺手将 39(UUID)、31(Battery)、50(Screen)、59(Remark) 一并加固

**方案二：有效绑定优先锁定（用户最终倾向的方案）**
- 引入 `_locked` 内部标记，字段一旦被有效绑定写入就锁定
- 解绑记录无法冲掉已锁定的绑定值
- 语义更强："只要存在有效绑定，无论如何保证取到值"
- 输出时自动剔除 `_locked` 内部标记

### 两版核心差异

| 场景 | 方案一（非空保护） | 方案二（有效绑定锁定） |
|---|---|---|
| 一有值一空 | ✅ 取到 | ✅ 取到 |
| 有效绑定 + 解绑混在一起 | ⚠️ 解绑后到会置 null | ✅ 锁定后解绑无效 |
| 空→有值→解绑顺序 | ⚠️ 最终 null | ✅ 仍稳取 |
