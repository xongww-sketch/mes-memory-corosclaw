## 🍎 Apple FindMy 产线对接方案讨论

**发起人**：熊旺
**目标**：基于最新 n8n Apple 项目工作流 + Apple 官方协议文档，设计一套长效稳定、覆盖维修场景的 FindMy Token 注册/激活/校验/下发方案。

### 当前现状梳理

- Apple 项目下有 6 个工作流，其中 2 个 ACTIVE（新 MES 版本）
- 数据流：产线扫码 → 申请 Apple Token(HTTP Request) → CRC16 转换 → Code 节点组装 → Postgres 插入 `machine_data` 表
- Token 状态流转：初始 → status=2（已使用）
- 新旧 MES 两套工作流并存，都在往 machine_data 写入

### 核心需求（熊旺提出的 5 个关键要求）

1. **唯一性校验**：FindMy 密钥序列号必须做唯一性，校验每个 token 长度和共性合法性，确保与历史数据库无重复
2. **写入后立即反查**：固件写入密钥后能立即通过 Apple 接口反查是否写入成功
3. **维修场景全覆盖**：主板被复用、主板 token 被替换、主板复用到新 SN 底壳上——所有换板/换件场景都需要方案覆盖
4. **激活状态可追踪**：某个 token 是否已被激活，能否被追踪到，以利于产线校验
5. **长效稳定、一次性解决**：方案要能长期运行，不用后续维护

### 关键业务细节确认

- **主板标识**：通过读取每块主板的 UUID 来区分主板
- **FindMy Token 来源**：通过硬件 NPI 从 MES 拉取信息存储
- **激活状态确认**：本地只有 token 状态机，是否被激活需查 Apple 官方文档，考虑是否有办法追踪 token 状态机以利于产线校验

### 维修场景分析（重点）

熊旺特别强调需要全面思考维修场景：
- 主板被写入 token 后若出现维修，主板被复用 → 是否需要重新下发 token？
- 主板 token 出现维修 → 是否需要重新下发？
- 主板被写过 token 且已激活 → 若主板复用到新的 SN 底壳上 → 能否被成功激活？

**待熊旺确认的方案细节**：
- Token 与主板 UUID 的绑定关系是否需要持久化到 MES
- 主板更换后的 token 重新下发流程
- 激活状态查询的权威数据源（Apple 官方 API vs 本地状态机）
